# RFC 2396 vs. RFC 3986: What Changed at the Obsoleting Boundary

RFC 3986 (January 2005) formally **obsoletes** RFC 2396. This page collects the concrete diffs — useful when a fact (e.g. "is `[` reserved?") depends on which generation of the spec is being read. See [[uri-vs-whatwg-url]] for how both compare against the modern WHATWG URL Standard.

## Grammar Notation

| | RFC 2396 | RFC 3986 |
|---|---|---|
| Notation | Informal BNF-like dialect | Formal **ABNF** (RFC 2234/4234), `/` for alternation |
| Top production | `URI-reference = [ absoluteURI \| relativeURI ] [ "#" fragment ]` | `URI-reference = URI / relative-ref` |
| URL/URN split | `absoluteURI = scheme ":" ( hier_part \| opaque_part )` — structural fork | Unified `URI = scheme ":" hier-part [ "?" query ] [ "#" fragment ]`, no opaque/hierarchical split at the grammar level (see [[rfc2396-grammar]], [[uri-generic-syntax]]) |

## Character Classes

| | RFC 2396 | RFC 3986 |
|---|---|---|
| Reserved set | One flat `reserved` set of 10 chars | Split into **gen-delims** (7: `: / ? # [ ] @`) and **sub-delims** (11: `! $ & ' ( ) * + , ; =`) — see [[uri-reserved-characters]] |
| Unreserved set | `alphanum \| mark`, where `mark` = 9 chars (`- _ . ! ~ * ' ( )`) | `ALPHA / DIGIT / "-" / "." / "_" / "~"` — only 4 of `mark`'s punctuation chars kept; `! * ' ( )` **reclassified as sub-delims** |
| IPv6 bracket chars | `[` `]` not in any grammar (no IPv6 in base 2396) | `[` `]` are gen-delims, used by `IP-literal` |
| "Unwise" category | Explicit `unwise = { } \| \ ^ [ ] \`` tier (discouraged, not forbidden) | Dropped — no soft "unwise" tier, just in-grammar or not |

See [[rfc2396-reserved-characters]] for full detail on the `mark` → sub-delims reclassification.

## Authority / Host

| | RFC 2396 | RFC 3986 |
|---|---|---|
| Authority | `authority = server \| reg_name` (two-way fork) | `authority = [ userinfo "@" ] host [ ":" port ]` (single uniform shape) |
| Host | `host = hostname \| IPv4address` (no IPv6 in base text; DNS-shaped `hostname`) | `host = IP-literal / IPv4address / reg-name` (three-way, includes `[...]` IPv6/IPvFuture) |
| IPv6 | Added later via **RFC 2732** update (`IPv6reference = "[" IPv6address "]"`) | Native, folded into `IP-literal` from the start |

See [[rfc2396-scheme-authority]] and [[rfc2396-host]].

## Path

| | RFC 2396 | RFC 3986 |
|---|---|---|
| Segment structure | `segment = *pchar *( ";" param )` — matrix-parameter sub-structure is part of the grammar | `segment = *pchar` — no `;param` subdivision; `;` is just a sub-delim |
| Path forms | `abs_path` / `rel_path` / `opaque_part` (opaque part skips hierarchy entirely for non-hier schemes) | Five mutually exclusive forms: `path-abempty` / `path-absolute` / `path-rootless` / `path-noscheme` / `path-empty`, all kept inside the unified hierarchical model |

See [[rfc2396-path]] and [[uri-path]].

## Reference Resolution

The core algorithm (scheme present → absolute; authority present → network-path; path starts with `/` → absolute-path; else → merge-and-remove-dot-segments) is **unchanged in logical structure** between the two RFCs. RFC 3986 reformats it as explicit pseudocode (§5.2.2–5.2.4: Transform References / Merge / Remove Dot Segments) instead of RFC 2396's numbered prose steps, and explicitly documents the `http:g` ambiguity as "a loophole in prior specifications" (i.e., in RFC 2396 itself) rather than silently inheriting it. See [[rfc2396-reference-resolution]] and [[uri-reference-resolution]].

## Terminology

RFC 2396 still implicitly carries the URL/URN distinction through its `hier_part`/`opaque_part` split. RFC 3986 §1.1.3 **explicitly deprecates "URL" and "URN" as formal terms**, recommending "URI" universally — a position statement RFC 2396 never made this directly. See [[uri-terminology]].

## Normalization

RFC 2396 mentions equivalence informally (case-insensitive scheme/host, default-port elision) without RFC 3986's structured §6.2.1–6.2.4 comparison ladder (string / syntax-based / scheme-based / protocol-based). See [[uri-normalization]].

## What Didn't Change

The five-component mental model (scheme, authority, path, query, fragment), the `%XX` escaping mechanism (`escaped` → `pct-encoded`, byte-identical behavior), the fragment-is-client-side-only rule, and the overall left-to-right hierarchical-reference philosophy all carry over from RFC 2396 into RFC 3986 essentially intact.
